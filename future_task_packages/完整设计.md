# 完整设计

> 当前激活路线：**安装器 + 本地 seller-Agent + 本地网页操作界面 + 服务器控制面 + 平台后端**。
> 这是一份面向最小 MVP 的总设计文档，不代表当前仓库所有模块都已实现。

## 1. 目标

项目目标不是先做成完整云厂商，而是先验证这样一条最小闭环：

- 买家像租阿里云 / 腾讯云机器一样，按小时租一台整机
- 卖家不需要理解 Docker、Swarm、WireGuard、CodeX、MCP
- 卖家只通过本地网页界面输入自然语言和点击按钮
- 本地 seller-Agent 负责解释意图、配置环境、接入网络、注册节点、推送镜像
- 平台后端只保存身份、节点、镜像、订单、计价、账单和收益真相
- Docker Swarm 只负责运行时编排

## 2. 当前仓库现状

当前仓库仍然是早期原型：

- `backend/` 只有最小后端骨架和一组平台接入接口
- `seller_client/` 只有本地 MCP 原型、安装器骨架和实验性界面代码
- 服务器 manager 已准备，`Portainer`、`registry` 已可用
- 还没有正式的平台前端
- 还没有正式的本地网页操作界面
- 还没有完整的订单、余额、账单、收益实现

## 3. 系统总览

最终系统按 5 个部分拆分：

1. 环境安装器
2. 本地 seller-Agent
3. 本地关联网页
4. 服务器端
5. 平台后端

## 5. MVP 决策

本轮最小 MVP 固定采用这些决策：

- 租用单位：**整机独占租用**
- 定价来源：**平台统一定价**
- 结算范围：**平台余额**
- 计费粒度：**按启动小时向上取整**
- 节点身份绑定：**卖家登录平台后，由后端签发节点令牌**
- 客户端交互方式：**网页只做操作和展示，不直接操作本机**
- 本机执行入口：**seller-Agent**
- 自然语言理解入口：**seller-Agent 内嵌 CodeX 能力**
- CodeX API key：**由平台后端签发 / 下发给 seller-Agent**

这意味着：

- 一个买家订单独占一台卖家节点
- 不做同机多租户切片
- 不做复杂桌面客户端壳子
- 不做纯公网网页直接操作本机
- 不让浏览器直接接触本机 Docker、WireGuard、文件系统

## 6. 五大部分职责

### 6.1 环境安装器

安装器只解决“卖家电脑很干净、什么都没装”的问题。

职责：

- 检测操作系统：`Windows` / `Linux`
- 检测是否已安装 `CodeX`
- 如果没有，安装或引导安装 `CodeX`
- 自动把本地 seller-Agent 暴露的 MCP 挂到 CodeX 配置
- 检测 / 安装 `Docker`
- 检测 / 安装 `WireGuard`
- 初始化本地工作目录、配置目录、日志目录
- 启动本地 seller-Agent，这里面是套壳code X ,在这个内部，APIkey  用平台的

明确不负责：

- 不保存业务状态真相
- 不理解订单与计费
- 不直接和买家交互

### 6.2 本地 seller-Agent

seller-Agent 是卖家机器上的核心执行器。

职责：

- 常驻本机运行
- 内嵌 CodeX 能力
- 对外暴露本地 MCP 工具
- 负责本机环境检测
- 负责 Docker 检查与拉起
- 负责 WireGuard 接入
- 负责节点注册与心跳
- 负责镜像构建、推送、登记
- 负责解释卖家自然语言意图
- 负责把复杂操作翻译成可执行步骤

明确不负责：

- 不持有订单真相
- 不持有账单真相
- 不自行决定价格

### 6.3 本地关联网页

本地关联网页是**操作界面**，不是执行器，用户在这里操作saller agent

职责：

- 登录展示
- 节点接入向导
- 状态查看
- 日志查看
- 镜像上传按钮和状态显示
- 接收卖家自然语言输入
- 将请求转发给本地 seller-Agent

关键约束：

- 页面本身不直接执行系统命令
- 页面不直接触碰 Docker、WireGuard、文件系统
- 页面只和本地 `localhost` 的 seller-Agent 通信

这意味着它更像：

- 本地 Web UI
- 或嵌入浏览器的本地操作页

而不是一个真正的系统执行层。

### 6.4 服务器端

服务器端是控制面基础设施，不是业务真相层。

职责：

- 唯一 `Swarm manager`
- 维护 `registry`
- 维护 `Portainer`
- 维护 `WireGuard` hub
- 管理节点加入、节点标签、运行时 service 落点

明确不负责：

- 不负责注册登录
- 不负责余额、账单、收益
- 不负责自然语言理解

### 6.5 平台后端

平台后端是身份和业务真相层。

职责：

- 注册 / 登录
- 卖家身份管理
- 节点令牌签发
- CodeX API key 签发或下发
- 节点注册 / 心跳
- 镜像元数据登记
- 节点和镜像状态查看
- 后续订单、余额、账单、收益

关键约束：

- 后端只保存状态与真相
- 后端不直接操作卖家电脑
- 后端不直接运行本机环境部署逻辑

## 7. CodeX API key 设计

### 原则

本地 seller-Agent 内嵌 CodeX，但 **CodeX API key 不由卖家手工填，也不写死在客户端里**。

固定规则：

- 由平台后端提供给 seller-Agent

### 最小流程

1. 卖家在本地网页登录平台
2. 平台后端签发：
   - `access_token`
   - `node_registration_token`
   - `codex_api_key` 或等价短期凭证
3. seller-Agent 把该 key 注入本地 CodeX 运行环境
4. 本地自然语言交互由 seller-Agent 调 CodeX 完成

### 为什么这样做

- 平台能统一控制配额和风控
- 卖家不需要理解 API key
- 可以后续做轮换和吊销

### MVP 要求

- 先支持后端签发单节点可用的 key
- 不要求一开始就做复杂轮换
- 但要给后续轮换留字段和接口位置

## 8. 自然语言接入目标

卖家可以说这样的话：

- “我能把自己电脑性能的 10% 上传到平台获取收益吗？”
- “我想把电脑闲置部分租出去，怎么弄？”
- “我不会 Docker，你帮我安装一下。”

seller-Agent 的最小行为应该是：

1. 识别这是卖家接入意图
2. 用小白能懂的话解释当前会发生什么
3. 检测本机环境
4. 安装或引导安装缺失组件
5. 接入 WireGuard
6. 登录平台并获取节点令牌
7. 注册节点并上报能力
8. 告诉卖家是否接入成功

注意：

- MVP 中“10% 性能”只作为卖家共享偏好，不要求立刻做硬件级精确切片

## 9. 最小买卖流程

### 9.1 卖家接入流程

1. 卖家运行安装器
2. 安装器准备：
   - CodeX
   - MCP 挂载
   - Docker
   - WireGuard
   - seller-Agent
3. 卖家打开本地网页
4. 在网页中输入自然语言接入意图
5. 页面把请求发给本地 seller-Agent
6. seller-Agent 解释并开始接入流程
7. 平台后端签发节点令牌与 CodeX API key
8. seller-Agent 完成节点注册和能力上报
9. 节点进入 `available` 或 `pending_review`

### 9.2 镜像接入流程

1. 卖家在本地网页发起镜像上传
2. 页面调用本地 seller-Agent
3. seller-Agent 构建镜像
4. seller-Agent 推送镜像到服务器 registry
5. seller-Agent 调平台后端登记镜像元数据
6. 平台后端将镜像与卖家节点、卖家身份建立关联

### 9.3 买家租用流程

1. 买家浏览平台页面上的可租节点
2. 平台展示统一小时价和节点摘要
3. 买家创建订单
4. 平台冻结首小时费用
5. 平台调用服务器控制面在目标节点创建 runtime service
6. 节点进入 `leased`
7. 订单进入 `running`

### 9.4 停止与结算流程

1. 买家主动停止或余额不足
2. 平台请求控制面销毁 runtime service
3. 平台结束 `LeaseSession`
4. 账单按已进入的计费小时数结算
5. 节点回到 `available`

## 10. Windows / Linux 策略

MVP 明确同时支持：

- `Windows`
- `Linux`

策略：

- 本地网页逻辑统一
- 平台后端接口统一
- seller-Agent 根据系统差异调用不同底层命令
- 安装器根据系统做不同依赖安装和服务注册

特别注意：

- Windows 优先处理：Docker Desktop、WireGuard for Windows、CodeX 配置
- Linux 优先处理：Docker Engine、`wg/wg-quick`、systemd 服务

## 11. MVP 不做什么

本轮明确不做：

- 不做同机多租户资源切片
- 不做分钟级计费
- 不做动态竞价
- 不做多币种
- 不做复杂退款与争议仲裁
- 不做买家自定义镜像上传
- 不做浏览器直接操作本机
- 不做真正的桌面壳应用

## 13. 一句话总结

最终路线不是“桌面 GUI 直接干所有事”，而是：

- **安装器准备环境**
- **本地 seller-Agent 执行复杂操作**
- **本地网页做操作界面**
- **服务器端做控制面**
- **平台后端做身份、状态与交易真相**

并且：

- **CodeX API key 由平台后端下发**
- **本地网页只是 seller-Agent 的操作界面**
